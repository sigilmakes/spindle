DATA FLOW FOR CODE EXECUTION
=============================

1. User calls: spindle_exec(code)
                       ▼
2. index.ts spindle_exec handler:
   • Sets currentOnUpdate, currentSignal, currentCode (closures!)
   • Calls repl.exec(code, signal)
                       ▼
3. repl.ts wraps code in async IIFE:
   (async () => { code })()
                       ▼
4. VM context executes with injected bindings
   ┌──────────────────────────────────────────┐
   │ Possible actions:                        │
   │                                          │
   │ a) Direct I/O:                          │
   │    await read({path})                   │
   │    await bash({command})                │
   │    ... (all wrapped by tools.ts)        │
   │                                          │
   │ b) Sub-agent orchestration:             │
   │    await llm(prompt, {agent?, model?})  │
   │      └─▶ agents.ts spawns pi process   │
   │      └─▶ streams events as JSON         │
   │      └─▶ returns final text + usage     │
   │                                          │
   │ c) Threaded task dispatch:              │
   │    const s1 = thread(task, opts)        │
   │    const s2 = thread(task, opts)        │
   │    const eps = await dispatch([s1,s2]) │
   │      └─▶ threads.ts work queue          │
   │      └─▶ up to 4 concurrent (configurable)
   │      └─▶ streams updates via closure    │
   │      └─▶ returns Episode[] array        │
   └──────────────────────────────────────────┘
                       ▼
5. Results formatted by render.ts
                       ▼
6. Displayed to user

CLOSURE STATE THREADING (Current Pattern - ⚠ Fragile)
======================================================

Per execution:

  spindle_exec(code, signal, onUpdate, ctx)
           │
           ├─► currentOnUpdate = onUpdate ◄──┐
           ├─► currentSignal = signal         │
           └─► currentCode = code             │
                                              │ Captured by closures
  initRepl() injects:                         │
                                              │
    r.inject({                                │
      dispatch: async (specs) => {            │
        const onDispatchUpdate = (states) => {
          if (!onUpdate) return;  ◄────────┘ Used here!
          currentOnUpdate?.({...});          │
        };                                    │
      }                                       │
    });                                       │


WHAT'S CLEAN
============

✓ Zero coupling between core modules (repl, tools, agents, threads)
✓ Single responsibility per file
✓ Clear type boundaries (ThreadSpec, Episode, ThreadState)
✓ Lazy async generator pattern for threading
✓ Work queue with simple concurrency control
✓ Event streaming from sub-agents
✓ No global state except REPL context (intentional)


WHAT COULD IMPROVE
===================

⚠ State threading via mutable closures (non-obvious data flow)
  Suggestion: Pass ExecContext object explicitly

⚠ Usage tracking scattered across multiple increments
  Suggestion: Centralize with UsageAccumulator class

⚠ Episode parsing fragile (regex on text)
  Suggestion: Add validation layer + better error messages

⚠ Error handling untyped (strings only)
  Suggestion: Define tagged error union types

⚠ Inconsistent abstractions in tools (text vs bulk I/O shapes)
  Suggestion: Document the distinction clearly (minor issue)

⚠ Mixed concerns in initRepl()
  Suggestion: Split into 4-5 focused setup functions
